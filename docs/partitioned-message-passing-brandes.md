# Partitioned Message-Passing Brandes

## 1) Контекст и проблема

Текущая параллельная версия (`BrandesBCParallel`) делит **источники** по задачам (`coforall`),
но каждая задача хранит полноразмерные массивы размера `n`:

- `dist[0..n-1]`
- `sigma[0..n-1]`
- `delta[0..n-1]`
- `queue[0..n-1]`
- `stack[0..n-1]`
- `localBC[0..n-1]`

Итог: память масштаба `O(numTasks * n)` (фактически несколько массивов `O(n)` на задачу).
При росте `numTasks` это становится основным bottleneck.

## 2) Новая идея

**Partitioned Message-Passing Brandes**:

- граф делится на `P` частей (partitions);
- каждая вершина имеет **ровно одного владельца** (owner part);
- часть хранит локальное состояние **только для своих вершин**;
- рёбра между частями обрабатываются через явные буферы сообщений.

Критически важно:

> Нельзя считать BC независимо в каждой части и потом суммировать.
> Это неверно.
> Нужно реализовать ту же глобальную рекурренцию Brandes,
> но доставлять межчастевые обновления через сообщения.

---

## 3) Модель владения вершинами

Пусть задана функция владения:

- `owner(v) -> partId in [0..P-1]`.

Требования:

1. Для каждой вершины `v` существует единственный `owner(v)`.
2. Для каждой части `p` храним множество `V_p = { v | owner(v)=p }`.
3. Множества `V_p` попарно не пересекаются, и их объединение равно всему `V`.

На первом этапе разбиение может быть простым (блочное по индексу вершины),
чтобы обеспечить воспроизводимость и простоту.

---

## 4) Локальные вершины, boundary/ghost

Для части `p`:

- **local vertices**: вершины `v in V_p` (полное локальное состояние для них);
- **boundary edges**: рёбра `(u,v)` где `owner(u) != owner(v)`;
- **ghost metadata**: минимальная служебная информация о внешних соседях
  (например, их global id + owner), без дублирования полного состояния графа.

Принцип:

- полные массивы `dist/sigma/delta` не храним для всех `n` внутри каждой части;
- храним их только для локальных `V_p`;
- межчастевые обновления приходят сообщениями и применяются к локальному состоянию владельца.

---

## 5) Forward BFS-фаза (RELAX messages)

Для фиксированного источника `s` выполняется уровень-за-уровнем BFS, но распределённо:

1. Инициализация:
   - `dist[s]=0`, `sigma[s]=1` у владельца `owner(s)`;
   - для остальных локальных вершин: `dist=INF`, `sigma=0`.
2. На уровне `d` каждая часть обрабатывает локальный frontier с `dist=d`.
3. Для ребра `(u,v)`:
   - если `owner(v)==owner(u)`: локальный relax;
   - иначе отправить `RELAX(v, candDist=d+1, sigmaContribution=sigma[u])`
     в буфер части `owner(v)`.
4. Получатель агрегирует `RELAX`:
   - если `candDist < dist[v]`: обновить `dist[v]`, сбросить `sigma[v]` на вклад сообщений с этим минимумом;
   - если `candDist == dist[v]`: добавить вклад в `sigma[v]`.

Важно: сообщения этого уровня должны быть полностью обработаны
до перехода к следующему уровню BFS.

---

## 6) Накопление sigma через границы частей

В обычном Brandes `sigma[v]` — число кратчайших путей от `s` до `v`.
В partitioned-варианте это сохраняется буквально:

- все вклады в `sigma[v]` с минимального расстояния приходят либо локально,
  либо через `RELAX` от других частей;
- владелец `v` суммирует все такие вклады;
- вклады с большего расстояния игнорируются.

Таким образом, `sigma` остаётся глобально корректной, хотя хранится распределённо.

---

## 7) Backward фаза (DEPENDENCY messages)

После завершения BFS известны уровни расстояний.
Далее идём по уровням в обратном порядке, как в Brandes:

- для вершины `w` нужно раздать вклад предшественникам `v`, где
  `dist[v] = dist[w]-1`.
- формула та же:
  \[
  \delta[v] += \frac{\sigma[v]}{\sigma[w]} (1 + \delta[w])
  \]

Если `v` и `w` в разных частях:

- вместо прямого обновления отправляется
  `DEPENDENCY(v, contributionFromW)` в часть `owner(v)`.

Владелец `v` аккумулирует входящие dependency-вклады.

Для `w != s` его `delta[w]` добавляется в `BC[w]` у владельца `w`.

---

## 8) Почему нужна синхронизация уровней BFS

Если перейти к уровню `d+1` до завершения обработки всех `RELAX` уровня `d`,
можно:

- пропустить более короткий путь, пришедший поздним сообщением;
- неверно посчитать `sigma` (не все вклады минимального уровня учтены).

Поэтому требуется **barrier по уровням BFS**:

- `compute local -> send -> receive/apply -> barrier -> next level`.

---

## 9) Почему нужна синхронизация уровней backward

`delta` на уровне `d` зависит от уже готовых `delta` на уровне `d+1`.
Если начать уровень `d`, пока часть сообщений с `d+1` ещё не доставлена,
рекурренция нарушится.

Поэтому в обратной фазе тоже нужен барьер по уровням:

- `process level L -> send dependency -> receive/apply -> barrier -> L-1`.

---

## 10) Эквивалентность обычному Brandes

Алгоритм эквивалентен обычному Brandes, потому что:

1. Используются те же инварианты `dist`, `sigma`, `delta`.
2. Используются те же формулы обновления.
3. Единственное отличие — **транспорт** межвершинных обновлений:
   - локально — прямой доступ,
   - межчастево — сообщение.
4. Барьеры уровней гарантируют тот же частичный порядок вычислений,
   который неявно есть в последовательном уровне-by-уровню алгоритме.

То есть это не аппроксимация, а иная организация той же математики.

---

## 11) Отличия от текущего `BrandesBCParallel`

Текущий `BrandesBCParallel`:

- делит источники между задачами;
- каждая задача хранит полноразмерные `O(n)` массивы;
- объединяет `localBC` в глобальный массив.

Partitioned message-passing Brandes:

- делит вершины между частями;
- хранит состояние только для локальных вершин каждой части;
- межчастевые зависимости и relax-обновления передаются сообщениями;
- синхронизация идёт по уровням BFS/backward.

Это принципиально другой параллельный decomposition.

---

## 12) Ожидаемое поведение памяти

При хорошем балансировании `|V_p| ~ n/P`:

- локальные состояния части `dist/sigma/delta/...` примерно `O(n/P)`;
- глобальная память по всем частям всё ещё `O(n)` на один источник
  (плюс буферы сообщений), а не `O(numTasks * n)` как в source-parallel подходе.

Что всё ещё может быть большим:

- буферы сообщений на уровнях с большим количеством boundary-рёбер;
- вспомогательные структуры для boundary/ghost metadata.

---

## 13) Ожидаемые коммуникационные расходы

Цена за снижение памяти:

- больше коммуникаций на межчастевых рёбрах;
- барьеры на каждом уровне BFS и backward;
- стоимость агрегирования сообщений (`RELAX` и `DEPENDENCY`).

Если граф «разрезан плохо» (много cross-partition edges),
коммуникационный overhead может доминировать.

---

## 14) Ограничения первой реализации

Первая версия должна быть консервативной:

1. **Single-process simulation** message passing (в пределах одного процесса/locale).
2. **Без реальной multi-locale коммуникации** на этом шаге.
3. **Correctness first, performance second**:
   - сначала полная эквивалентность Brandes,
   - потом оптимизации буферов/расписания/баланса.

---

## 15) План внедрения (высокоуровнево)

1. Новый модуль (например, `BrandesBCPartitioned`).
2. Явная структура partition metadata:
   - owner(v), local vertex lists, boundary adjacency info.
3. Реализация forward level-synchronous с `RELAX` буферами.
4. Реализация backward level-synchronous с `DEPENDENCY` буферами.
5. Проверка против Naive/Seq Brandes на малых и случайных графах.
6. Интеграция в `Run` как экспериментального режима (после подтверждения корректности).


## 16) Дополнительные пояснения для защиты/обсуждения

### Почему local BC per partition недостаточен

BC зависит от всех кратчайших путей в графе, а не только от подграфа partition.
Если путь пересекает границу partition, его вклад в `sigma` и `delta` должен пройти через владельцев вершин на границе.
Без этого итоговая центральность будет занижена/искажена.

### Почему cross-part shortest paths требуют сообщений

При модели единственного владельца состояния вершины только owner-part может менять `dist/sigma/delta` этой вершины.
Значит межpartition обновления обязаны идти в виде сообщений (`RELAX` и `DEPENDENCY`),
а не через прямой доступ к чужому локальному состоянию.

### Questions a teacher may ask

- **Why is this correct?**  
  Потому что используются исходные уравнения Brandes и level-synchronous порядок,
  а сообщения только реализуют доставку межpartition обновлений к владельцам.

- **Why not just sum local BC values?**  
  Это игнорирует межpartition кратчайшие пути и dependency-вклады через границы.

- **What data is owned by each partition?**  
  `dist/sigma/delta` и frontier для owned-вершин, плюс локальные буферы входящих сообщений.

- **What messages are sent?**  
  `RELAX(targetVertex, distance, sigmaContribution)` в forward-фазе,
  `DEPENDENCY(targetVertex, contribution)` в backward-фазе.

- **Where is synchronization required?**  
  Между уровнями BFS и между уровнями backward dependency.

- **What is the main memory advantage?**  
  Переход от `O(numTasks * n)` временных структур к `O(local_n)` на partition (+ buffers).

- **What is the main performance drawback?**  
  Коммуникации и барьеры, особенно при большом числе cut-edges.
